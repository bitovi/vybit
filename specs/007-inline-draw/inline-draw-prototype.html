Inline Drawing Canvas — UX Exploration

Prototyping how users insert a drawing canvas into the page flow, the drawing tools available, resizing behavior, and how sketches become queued patches for an AI agent to implement.

1 Insert Control — Draw Button on Selection Highlight

When an element is selected (teal highlight), a small pencil button appears anchored to the selection box. Clicking or hovering reveals a menu to choose insert position: Before, After, First child, or Last child.

Recommendation: Place the draw button on the top-right corner of the selection highlight. This position is:
  • Close to the existing container switcher button
  • Out of the way of content
  • Consistent with other floating controls in dev tools
Hover vs click: The menu should appear on click (not hover) to avoid accidental reveals while selecting elements. A click is intentional; hover would be annoying during normal inspection.
2 Drawing Frame — Injected Inline in Page Flow

After choosing an insert position, a dashed-border frame appears in the page DOM as a sibling/child. It contains a toolbar, a drawing area (Fabric.js canvas), and a footer with actions. The frame is resizable by dragging the bottom edge or corner.

Styling: 2px dashed teal border distinguishes the canvas from page content. Inline styles only (no class conflicts with target page).
Resize: Drag bottom edge (↕), right edge (↔), or corner (↘) to resize. Min 200×150px. Width defaults to the selected element's width.
Footer: Shows insert position context. "Queue as Change" captures the canvas as a PNG + element context and adds it to the draft queue. "Close" removes the frame.
3 Child Insertion — Canvas Inside the Selected Element

When "First child" or "Last child" is chosen, the canvas is injected inside the element rather than as a sibling. This is useful when designing new content within a container (e.g. adding a new card to a grid, a new row to a list).

4 Panel — Draw Tab & How Drawing Patches Appear in the Queue

After "Queue as Change", the drawing becomes a patch in the queue — visible in the panel's footer and in the Draw tab. The patch carries a thumbnail preview of the sketch, the target element context, and the insert position. The AI agent receives this as a multi-part message with the image inline.

Patch types: The queue now holds two types of patches:
  • Drawing — base64 PNG image + element context + insert position
  • Classes — existing Tailwind class change patches
MCP delivery: When the agent calls implement_next_change, drawing patches are returned as multi-part content — an image block (PNG) alongside a text block with the element context, component name, and instructions for what to build. This lets vision-capable models "see" the sketch.

5 Full Flow — Step by Step
1
Click element on page
Teal selection highlight appears. Pencil (✎) button shows on the highlight corner.
2
Click the pencil button
Position menu opens: Before, After, First child, Last child
3
Choose insert position
Drawing frame appears in the page flow at the chosen position. Toolbar, canvas, and footer render inline.
4
Draw your design
Use freehand, shapes, text, colors. Resize the frame by dragging edges/corners. Undo/redo as needed.
5
Click "Queue as Change"
Canvas → PNG → queued as a draft drawing patch. Thumbnail shows in the Draw tab. Drawing frame stays open for edits.
6
Commit the change
Commit from the queue footer (same flow as class changes). Drawing patch transitions to "committed" status.
7
Agent receives multi-part content
MCP implement_next_change returns: image block (PNG of sketch) + text block (component name, DOM context, insert position, instructions). Vision-capable agents "see" what to build.
6 Color Palette — In the Drawing Toolbar

Clicking the fill or stroke color button opens a small inline palette. Starts with basic drawing colors. In the future, the palette will be populated from the user's Tailwind theme (their app's color tokens) — but a sensible default set works for v1.

  • Bottom edge — vertical resize (ns-resize cursor)

  • Default width: match selected element's width
  • Default height: 300px
8 Agent Delivery — Multi-Part MCP Content

When the agent calls implement_next_change and the next change is a drawing, the MCP response includes both image and text content blocks. 

{
  "content": [
    {
      "type": "text",
      "text": "## Design Request\n\nThe user has drawn a design sketch for a new UI element.\n\n**Component:** ProductCard\n**Target:** <div class=\"flex flex-col gap-4 p-6\">\n**Position:** Insert AFTER the target element (as sibling)\n**Canvas size:** 480×200px\n\n**Context (surrounding DOM):**\n```html\n<section class=\"grid grid-cols-3\">\n  <div class=\"flex flex-col gap-4 p-6\"> <!-- TARGET -->\n    ...\n  </div>\n</section>\n```\n\nPlease implement what the user has sketched in the attached image. The sketch shows the intended layout and visual hierarchy. Create appropriate React JSX with Tailwind classes.\n\nAfter implementing, call mark_change_implemented."
    },
    {
      "type": "image",
      "data": "iVBORw0KGgoAAAANSUhEUgAA...",
      "mimeType": "image/png"
    }
  ]
}
Why multi-part? MCP content arrays support mixed types. By sending the sketch as an image content block (not base64 in a text string), vision-capable models like GPT-4o or Claude can directly "see" the drawing. The text block provides structured context the model needs to know where in the code to implement it.
